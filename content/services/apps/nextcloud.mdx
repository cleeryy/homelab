## Placement

- VM `200` `ubuntu-nextcloud` on `pve1`.
- Beszel label `Nextcloud`, IP `192.168.1.27`.
- OS line reported by Beszel as `6.8.0-111-generic` (Linux kernel string).

## Docker containers

| Container | Function |
|-----------|----------|
| `nextcloud-app` | Nextcloud application |
| `nextcloud-db` | Database for Nextcloud |
| `nextcloud-cron` | Background job runner |
| `nextcloud-redis` | Cache / locking backend |
| `collabora` | Collabora Online (document editing) |
| `zerobyte` | Application present on the host; specific role not documented |
| `portainer` | Container management UI |
| `beszel-agent` | Monitoring agent |

`portainer` was reported as **restarting** at the last observation. Cause not investigated.

## Dependencies

- The Nextcloud stack is self-contained: database, cache, and document editor all run on the same host.
- Ingress goes through the reverse-proxy layer on `pve1` — exact route not captured here.
- Storage paths (whether user files live on the local VM disk, on `nas`, or on a TrueNAS share) are **not captured here**.

## Operational notes

- Stack uses standard Nextcloud images with a sidecar for cron. Restart order matters: `nextcloud-db` and `nextcloud-redis` come up before `nextcloud-app`.

## Failure impact

- A `pve1` outage takes the Nextcloud host with it.
- Loss of `nextcloud-db` data is the most critical risk; backup of this volume is essential and not documented here.
