## Placement

- VM `302` `ubuntu-qbittorrent` on `pve3`.
- Beszel label `QBittorrent`, IP `192.168.1.28`.

## Docker containers

| Container | Function |
|-----------|----------|
| `qbittorrent` | Download client |
| `gluetun` | VPN client wrapping the download traffic |
| `portainer` | Container management UI |
| `watchtower` | Image auto-update |
| `beszel-agent` | Monitoring agent |

## Dependencies

- Downloads land on a shared volume consumed by the Servarr stack (`301`) and read by Jellyfin (`300`). The exact mount path on TrueNAS is **not captured here**.
- `qbittorrent` is expected to share `gluetun`'s network namespace so that download traffic exits through the VPN; the exact compose wiring is not exported here.

## Operational notes

- `gluetun` was reported as **unhealthy** at the last observation. This is the most common single failure for this host: when `gluetun` is unhealthy, `qbittorrent` either has no network or no internet egress depending on the compose mode.

## Failure impact

- A `pve3` outage takes qBittorrent, the Servarr stack, Jellyfin, and TrueNAS down together.
- Loss of `gluetun` health stops downloads but leaves the rest of the media stack working from already-downloaded content.
