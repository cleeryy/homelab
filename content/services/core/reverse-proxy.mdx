## Components

Two reverse-proxy VMs are visible on `pve1`:

| VM ID | Name | Function |
|-------|------|----------|
| `100` | `ubuntu-nginx-proxy-manager` | Nginx Proxy Manager |
| `105` | `ubuntu-nginxrpm` | Second reverse-proxy VM |

In addition, an outbound tunnel runs as an LXC:

| CT ID | Name | Function |
|-------|------|----------|
| `103` | `cloudflared` | Outbound tunnel for selected services |

## Placement

All three components live on `pve1`.

## Known facts

- Both reverse-proxy VMs are running on the management LAN (`192.168.1.0/24`).
- The user historically used an Apache-based reverse proxy as well; this documentation focuses on the components that are currently deployed.
- Per-host certificates, ACME providers, and DNS validation are not exported into this documentation.

## Operational notes

- Ingress is the entry point for user-facing services; restarts must be planned with care.
- A single canonical ingress path is **not confirmed**. Either `100` or `105` could be canonical, or each could front a different subset of services. This is tracked in the [migration plan](/architecture/migration-plan).

## Failure impact

- If both reverse-proxy VMs are down, all HTTPS-fronted services on the management LAN become unreachable from outside the LAN.
- If `cloudflared` is down, services routed through the tunnel become unreachable from outside.
- OPNsense (`401` on `pve3`) is the WAN/router layer and is upstream of all of the above; its failure has broader impact.
