## Purpose

Summarise the homelab as a single coherent system: which nodes exist, what each one is for, and how workloads, storage, and traffic flow between them.

## Scope

A map of the platform, not a manual. Each topic links to its dedicated page.

## Platform

- Two-node Proxmox VE 9.2 cluster named `home`, with quorum enabled.
- `pve3` (`192.168.1.103`) — primary compute and production node.
- `pve1` (`192.168.1.101`) — infrastructure and management node.
- `pve2` is retired in place; it remains visible in some Proxmox views but is not part of the active design.

The cluster is **not symmetric**. `pve3` is the stronger node and carries the bulk of execution; `pve1` is the stable host for shared platform services. High availability across nodes is not in scope.

## Building blocks

- **Edge / routing.** OPNsense runs as a VM on `pve3` and bridges the management LAN (`vmbr0`) with the routed internal network on `vmbr1`.
- **Ingress.** Nginx Proxy Manager on `pve1` terminates HTTPS for user-facing services. A `cloudflared` LXC provides outbound tunneling for selected services.
- **Source of truth for code.** Forgejo on `pve1`, with a Forgejo runner on `pve3` for CI compute.
- **Documentation.** This Fumadocs site is becoming the canonical documentation surface, replacing Docmost (transitional).
- **Storage.** TrueNAS on `pve3` owns the bulk and shared data tier via passed-through disks. A ZFS pool on `pve1` (`pve1-storage`) backs infra-side data. An NFS share (`nas`) is mounted on both nodes for shared assets.
- **Observability.** Beszel collects host metrics across the cluster.
- **Workloads.** Apps, media stack, and lab workloads live on `pve3`.
