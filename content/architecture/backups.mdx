## Configured destinations

| Destination | Type | Where | Content allowed | Policy |
|-------------|------|-------|-----------------|--------|
| `local` | Directory on each node | `/var/lib/vz` on `pve1` and `pve3` | `backup`, `iso`, `vztmpl` | Default Proxmox retention |
| `nas` | NFS | `192.168.1.16:/data/pve` | `rootdir`, `iso`, `backup`, `images` | `prune-backups keep-all=1` |

`prune-backups keep-all=1` on `nas` means the configured pruning does not delete any backups. Retention on `local` is whatever the per-job Proxmox configuration sets — **not captured here**.

## Snapshot capabilities

- **ZFS snapshots** on `pve1-storage` (`pve1`) for infra-side application data.
- **TrueNAS snapshots** for datasets backing media and bulk data on `pve3`.

Snapshots are a recovery feature, not an off-host backup.

## Unknown / pending

- Per-VM and per-LXC backup schedule (cadence, retention).
- Whether `vzdump` runs against `nas`, `local`, or both for each guest.
- Whether TrueNAS replicates datasets to an off-site or off-chassis target.
- Whether the Docker hosts' application data (Nextcloud `nextcloud-db`, n8n state, Servarr config, qBittorrent state) is backed up by an additional path on top of the VM-level `vzdump`.

These items are not documented in the source material and are not guessed at here. See [Operations > Backup & restore](/operations/backup-restore) for procedures.

## Practical implications

- The `nas` NFS share is the only off-host backup target visible in the storage configuration. It is hosted on `192.168.1.16`, which is on the same management LAN as the rest of the cluster — it is not on a separate failure domain.
- The TrueNAS VM lives on `pve3`. A `pve3` failure affects both the production workload and a large fraction of the storage layer.
