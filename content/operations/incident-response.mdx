---
title: Incident response
description: Triage and recovery procedure when something breaks.
---

## Triage

- [ ] Identify the affected layer: Proxmox host, VM/LXC, Docker container, or external dependency.
- [ ] Check Beszel for host and container-level state. Recent observations to expect as benign:
  - `gluetun` on `QBittorrent` may be unhealthy without taking the whole stack down.
  - `portainer` on `Nextcloud` may be in a restarting loop without taking Nextcloud itself down.
  - `jellyfin` memory usage can be high during transcoding.
- [ ] Locate the service on [Service placement](/architecture/service-placement) to know which node and which dependencies are involved.

## Mitigate

- [ ] Apply the smallest change that restores service: restart the container, restart the guest, or restart the host service.
- [ ] If WAN is lost: check OPNsense (VM `401` on `pve3`).
- [ ] If ingress is lost: check `100` `ubuntu-nginx-proxy-manager`, `105` `ubuntu-nginxrpm`, and `103` `cloudflared` (all on `pve1`).
- [ ] If storage is unreachable: check `truenas` (`127`) and the `nas` mount on each affected node.

## Recover

- [ ] If a backup restore is needed, follow [Backup & restore](/operations/backup-restore).
- [ ] Validate end-to-end before declaring resolution. Beszel state is a fast smoke test, not a validation.

## Follow up

- [ ] Short postmortem: what happened, what was done, what changed.
- [ ] Update affected documentation (service page, architecture page, this page) so the next incident is shorter.

## Useful constraints

- Both reverse-proxy VMs live on `pve1`. A `pve1` host outage takes ingress down even if `pve3` is healthy.
- TrueNAS lives on `pve3`. A `pve3` host outage takes the media stack, lab containers, and a large portion of storage with it.
- OPNsense lives on `pve3`. A `pve3` host outage takes WAN with it.
