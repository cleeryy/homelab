## Configured destinations

- `local` (`/var/lib/vz`) on each node — accepts `backup`, `iso`, `vztmpl`.
- `nas` (`192.168.1.16:/data/pve`, NFS) — accepts `rootdir`, `iso`, `backup`, `images`. `prune-backups keep-all=1`.

## Known building blocks

- **Proxmox `vzdump`** for VM and LXC archives.
- **ZFS snapshots** on `pve1-storage` for infra-side application data.
- **TrueNAS snapshots** for the datasets backing media and bulk data.
- **Docker volumes** for application state on the Docker host VMs — captured by `vzdump` of the host VM, not by a Docker-aware backup.

## Backup procedure (manual)

### `vzdump` for a VM or LXC

- [ ] Pick the target storage (`local` or `nas`).
- [ ] Trigger from the Proxmox UI or `vzdump <vmid> --storage <storage>` on the node hosting the guest.
- [ ] Confirm the archive landed and its size is consistent with previous runs.

### ZFS snapshot on `pve1-storage`

- [ ] Take a snapshot of the relevant dataset.
- [ ] Consider sending the snapshot off-host (target **pending**).

### TrueNAS dataset

- [ ] Snapshot the dataset from the TrueNAS UI.
- [ ] Consider TrueNAS-native replication to an off-chassis target (**pending**).

Schedules and retention values are **not captured here**. Only `prune-backups keep-all=1` on `nas` is documented (no pruning).

## Restore procedure

### Single VM / LXC from `vzdump`

- [ ] Identify the archive (date, source storage, checksum if available).
- [ ] On the destination node: `qmrestore` for VMs, `pct restore` for LXCs.
- [ ] If the guest holds application data on a separate volume (e.g. `pve1-storage` dataset, TrueNAS share), restore that volume from its snapshot.
- [ ] Boot and validate end-to-end before swapping ingress / DNS.

### Docker host application data

- The `vzdump` archive of the host VM captures whatever lives inside the VM at the time of backup, including Docker volumes (assuming they live on the VM disk).
- Application-specific dumps (e.g. `nextcloud-db` SQL dump) are **not** automated by this procedure. Per-service dump steps belong on each service page.

### Whole-node failure

#### `pve1` down

- [ ] Stand up replacement `pve1` (or repair it).
- [ ] Restore ingress first: `100` Nginx Proxy Manager, `105` `ubuntu-nginxrpm`, `103` `cloudflared`.
- [ ] Restore `forgejo` (`111`) and `postgresql` (`112`).
- [ ] Restore `beszel` (`202`).
- [ ] Restore remaining platform services.

#### `pve3` down

- [ ] Stand up replacement `pve3` (or repair it).
- [ ] Restore `opnsense` (`401`) first if WAN is lost.
- [ ] Restore `truenas` (`127`) and re-attach the passed-through disks; verify pool import on the TrueNAS side.
- [ ] Restore media stack (`300`, `301`, `302`).
- [ ] Restore lab and supporting LXCs.

## Test cadence

Restore drills are **not documented**. At least one workload per tier should be exercised; the cadence is to be defined.
