## Decide where it goes

- **Application Docker stack** → an Ubuntu VM on the appropriate node, following the pattern used for Nextcloud/n8n/Servarr (Docker compose + `portainer` + `watchtower` + `beszel-agent`).
- **Single-purpose system service** → Proxmox LXC.
- **Storage-adjacent workload** → coordinate with TrueNAS for dataset and share creation.
- **Lab / pre-production** → `pve3`, in a clearly separated namespace.

## Allocate the ID

ID ranges currently in use are documented in [Inventory > Virtual machines](/inventory/virtual-machines). Pick a free ID in the matching range.

## Provision the guest

### VM (Docker host)

- [ ] Create on the target node.
- [ ] OS: Ubuntu 24.04 LTS to stay consistent with the rest of the Docker hosts.
- [ ] Install Docker and the compose plugin.
- [ ] Install the `beszel-agent` container and register the host in Beszel.
- [ ] Install `portainer` and `watchtower` if the host pattern uses them.
- [ ] Drop the compose file under version control (Forgejo).

### LXC

- [ ] Create on the target node from a vetted template.
- [ ] Apply baseline configuration (timezone, networking, SSH).
- [ ] Add to Beszel monitoring if relevant.

## Wire it in

- [ ] If user-facing, add a route through the reverse-proxy layer on `pve1`.
- [ ] If the service stores data, decide where the data volume lives (`local-lvm`, `pve1-storage`, TrueNAS dataset, `nas`).
- [ ] If the service has a backend (Postgres, Redis), confirm the backend exists or stand it up first.

## Document it

- [ ] Add or update the matching page under [Services](/services/core/reverse-proxy).
- [ ] Update the placement table in [Service placement](/architecture/service-placement).
- [ ] Update the inventory page for VMs or LXCs.
