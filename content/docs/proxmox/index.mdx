# Proxmox

## Cluster

| Node | IP | Role | Status | Version |
|------|----|------|--------|---------|
| PVE-01 | 10.0.10.2 | Principal | 🟢 Online | Proxmox VE 8.2 |
| PVE-02 | 10.0.10.3 | Secondaire | 🟢 Online | Proxmox VE 8.2 |

Le cluster est formé entre les deux nœuds sur le VLAN Management (10). La communication inter-nœuds passe par une interface réseau dédiée.

**Adresse du quorum** : PVE-01 (premier nœud, qdevice non utilisé — cluster à 2 nœuds avec paramètre `expected: 1`).

## Stockage partagé

### Défini dans Proxmox

| Storage ID | Type | Content | Serveur | Description |
|-----------|------|---------|---------|-------------|
| nas-backups | NFS | vzdump, iso | TNAS-01 — tank-01/backups | Sauvegardes VM/CT |
| nas-iso | NFS | iso, vztmpl | TNAS-01 — tank-01/templates | Modèles ISO et CT |
| nas-vms | iSCSI | images, rootdir | TNAS-01 — tank-01/vms | Disques VM |
| nvme-fast | iSCSI | images | TNAS-01 — nvme-pool/vms | Disques VM haute perf |
| local | Directory | all | Local (rpool) | Stockage local du nœud |

### Stratégie de placement

- **VM critiques** → stockage local (rpool) + réplication vers l'autre nœud
- **VM standards** → iSCSI sur NAS (tank-01/vms ou nvme-pool/vms)
- **Sauvegardes** → NFS vers NAS (tank-01/backups)
- **Templates** → NFS vers NAS (tank-01/templates) — partagé entre les nœuds via un bind mount local ou NFS direct

## Réseau Proxmox

| Bridge | VLANs | Usage |
|--------|-------|-------|
| vmbr0 | Native (VLAN 1) | Réseau de management initial |
| vmbr1 | Trunk (10, 20, 30, 40, 50, 99) | VLAN-aware bridge pour VM/CT |
| vmbr2 | VLAN 20 | Dédié au stockage iSCSI / NFS |

**Configuration lien** : NIC1 → vmbr0 + vmbr1 (bonding si plusieurs NIC disponibles), NIC2 → vmbr2 (dédié storage).

## Réplication

| VM/CT | Source | Destination | Mode |
|-------|--------|-------------|------|
| gw-01 | PVE-01 | PVE-02 | Ha disponibilité (HA) |
| svc-docker-01 | PVE-01 | PVE-02 | Réplication horaire |
| svc-db-01 | PVE-02 | PVE-01 | Réplication horaire |

**HA configurée** : seule GW-01 est en haute disponibilité avec démarrage automatique sur l'autre nœud en cas de panne.

## Quorum et maintenance

- Cluster à 2 nœuds : `expected` votes = 1 (pour éviter le split-brain)
- Arrêt d'un nœud : `pvecm expected 1` avant shutdown si nécessaire
- Migration à chaud possible pour les VM sur stockage local via réplication
